## 二、技术架构（已实现）

```
小程序3入口 → H5统一页面 → FastAPI后端 → 对话处理流程
                                          │
                    ┌─────────────────────┼──────────────────────┐
                    ▼                     ▼                      ▼
              火山方舟大模型          MySQL数据库            转人工工单
         (意图分类/知识检索)     (知识库/会话/运营数据)      (待接七鱼)
```

### 对话处理流程（实际运行）

```
用户消息
  │
  ├─ ① 多轮上下文检查
  │    └─ 上一轮追问品牌 + 本次含品牌 → 直接返回该品牌知识
  │
  ├─ ② 快速通道（不调大模型）
  │    ├─ 转人工/高风险词/超出范围 → 转人工
  │    └─ 问候语 → 欢迎语+FAQ卡片
  │
  ├─ ③ 意图分类（豆包大模型）
  │    └─ knowledge_query / live_query / unknown
  │
  ├─ ④ 知识检索（knowledge_query）
  │    ├─ L1 精确匹配（SQL，高分快速通道）
  │    ├─ L3 大模型全量检索（主路径，含主题标签）
  │    └─ L2 关键词兜底
  │
  ├─ ⑤ 品牌感知
  │    ├─ 命中need_brand=1 + 已指定品牌 → 返回该品牌知识
  │    ├─ 命中need_brand=1 + 未指定品牌 → 追问品牌（存pending）
  │    └─ need_brand=0 → 直接返回通用答案
  │
  └─ ⑥ 查询意图（live_query）→ 追问VIN/终端号（待接运营数据查询）
```

## 三、关键技术决策与实现

### 1. 知识检索策略：大模型全量检索为主

**演进过程**：
- v1：L1精确→L2关键词→L3向量（经典三层）
- v2：加rerank修正（无效，召回阶段错了排序救不回）
- v3（当前）：**大模型全量检索为主**

**为什么**：144条规模下，大模型直接从全量选最相关，比"关键词召回+精排"更准更简单，避开通用词干扰（"设备""怎么办"等词把无关条目顶到前面）。

**实现**：
- L1 精确匹配（标准问题完全相等/问法变体LIKE）→ 高分快速通道，不调大模型
- L3 大模型全量检索 → 把144条知识（ID+主题+标准问题）全塞给豆包，选top5
- L2 关键词匹配 → L3失败时兜底，保证服务可用

**实测效果**：

| 问题 | 命中 |
|---|---|
| 怎么查SIM卡号 | 查询SIM/ID方法主题（极目+极目单北斗+航天3条）✅ |
| 设备怎么重启 | 按键重启方法主题（3条）✅ |
| 我的定位不好使 | 不定位时屏幕显示主题（3条）✅ |
| 如何导出视频 | 通用知识KB0144（need_brand=0，直接答）✅ |

### 2. 品牌识别追问机制

**问题**：104条need_brand=1的知识按6个品牌分（极目/极目单北斗/航天/雅迅/启明/有为），不同品牌答案不同。

**实现**：
- 命中need_brand=1知识时，若用户未指定品牌 → 追问"您的设备是什么品牌"，列出6个候选
- 用户选品牌后 → 用`category_l2`(主题)+品牌查该品牌对应知识返回
- **多轮上下文衔接**：追问时把`{type:"brand_collection", knowledge_id, category_l2}`存入会话metadata，下一轮用户回答品牌时跳过意图分类直接返回该品牌知识

**实测**：
```
用户: 怎么查SIM卡号
系统: 关于「查询终端号/SIM卡号/ID号」...请问您的设备是什么品牌?
用户: 雅迅
系统: [返回雅迅查询SIM/ID方法答案] ✅
```

### 3. 转人工决策引擎

5种触发条件（确定性规则，不依赖大模型）：
1. `consecutive_fail`：连续3轮未解决
2. `keyword`：高风险词（投诉/退款/315/12315/起诉/维权）
3. `user_request`：用户明确要求转人工
4. `out_of_scope`：超出AI范围（修改绑定/激活设备/修改套餐）
5. `risk`：知识条目标记high风险

非工作时间 → 引导用户留问题描述+联系方式，自动生成工单。

### 4. 大模型的作用边界

大模型只做**理解**，不做**生成**：

| 环节 | 谁负责 | 大模型参与 |
|---|---|---|
| 意图分类 | 豆包大模型 | ✅ |
| 知识检索（选最相关） | 豆包大模型 | ✅ |
| 知识回答内容 | 知识库standard_answer原文 | ❌ |
| 数据库查询 | 精确SQL | ❌ |
| 转人工判断 | 关键词规则 | ❌ |

符合架构铁律：运营数据不直接给大模型，知识回答用标准答案而非大模型生成。
