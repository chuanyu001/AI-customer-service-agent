## Context

本方案基于《小程序AI客服——产品功能需求6.21》和四项业务知识库（行车记录仪/WiFi/流量/加油），构建覆盖四大业务领域的 AI 智能客服 Agent。

**核心约束**：
- 本地 MySQL + 火山方舟大模型（豆包），H5 页面不直连大模型和数据库
- 运营平台数据通过接口实时查询（batchVehicleInfo），不落库、不直接给大模型
- 四业务知识库已全部导入数据库（dashcam 144 / wifi 9 / data 11 / refueling 8 / youwei_device 10010）
- 每条知识标记「是否对客」（对客/转人工），转人工类知识命中后先追问收集信息再转人工，转人工工单携带完整对话上下文给坐席

**当前实际进度**：四业务知识库全量导入 + 向量语义检索 + 规则化意图识别已上线；正常主路径不再每条消息调用大模型；行车记录仪业务全链路跑通，前端 H5 可对话。

              本地向量模型          MySQL数据库            转人工工单

### 对话处理流程（实际运行）

```
用户消息
  │
  ├─ ① 多轮上下文检查
  │    └─ 上一轮追问品牌 + 本次含品牌 → 直接返回该品牌知识
  │
  ├─ ② 快速通道（不调大模型）
  │    ├─ 转人工/高风险词/超出范围 → 转人工
  │    └─ 问候语 → 欢迎语+FAQ卡片
  │
  ├─ ③ VIN主动品牌识别 (NEW: dashcam业务, 纯VIN消息)
  │    └─ 主动查VIN+终端号双因子→识别到品牌→返回+存pending
  │
  ├─ ④ 意图识别（规则优先）
  │    └─ business_area + knowledge_query / live_query / transfer_request / greeting
  │
  ├─ ⑤ 知识检索（knowledge_query）
  │    ├─ L1 精确匹配（SQL，高分快速通道）
  │    ├─ L2 本地向量召回（正常主路径）
  │    └─ L3 低置信候选重排（可选LLM，仅topK候选）
  │
  ├─ ⑥ 品牌感知
  │    ├─ 命中need_brand=1 + 已指定品牌 → 返回该品牌知识
  │    ├─ 命中need_brand=1 + 未指定品牌 → 追问品牌（存pending）
  │    └─ need_brand=0 → 直接返回通用答案
  │
  ├─ ⑦ 是否对客判断
  │    ├─ auto_reply=True（对客） → 直接返回知识答案
  │    └─ auto_reply=False（转人工）→ 用transfer_prompt追问收集信息（存pending=transfer_collection）
  │         └─ 下一轮用户回应 → 提取VIN/ICCID → 转人工，工单带完整上下文
  │
  └─ ⑧ 查询意图（live_query）→ 提取VIN → 调运营平台接口查询 → 返回结果
```

## 三、关键技术决策与实现

### 1. 知识检索策略：本地向量为主，大模型低频兜底

**演进过程**：
- v1：L1精确→L2关键词→L3向量（经典三层）
- v2：加rerank修正（无效，召回阶段错了排序救不回）
- v3：大模型全量检索（准确但慢、成本高）
- v4（当前）：**精确匹配 + 本地向量召回为主，大模型只在低置信topK候选内重排**

**为什么**：四业务知识库已达到172条，继续把全量知识发给大模型会带来3-5秒额外延迟和token成本。当前方案把知识向量预计算到 `business_knowledge_embedding`，启动加载到内存，正常问题只做本地向量化和余弦召回。

**实现**：
- L1 精确匹配（标准问题完全相等/问法变体LIKE）→ 高分快速通道，不调大模型
- L2 本地向量召回 → 用户问题用同一个本地模型生成向量，与内存向量库做余弦相似度
- L3 低置信LLM候选重排 → 只把topK本地候选交给大模型，不再传全量知识库
- 关键词不再作为普通答案兜底，只用于转人工、业务域路由、意图提示等规则判断

**实测效果**：

| 问题 | 命中 |
|---|---|
| 怎么查SIM卡号 | 查询SIM/ID方法主题（极目+极目单北斗+航天3条）✅ |
| SIM卡插在哪里 | SIM卡拔插方法 ✅ |
| 终端编号在哪里查 | 查询SIM/ID方法 ✅ |
| 记录仪服务到期后怎么处理 | 设备续费知识 ✅ |
| 设备怎么重启 | 按键重启方法主题（3条）✅ |
| 我的定位不好使 | 不定位时屏幕显示主题（3条）✅ |
| 如何导出视频 | 通用知识KB0144（need_brand=0，直接答）✅ |

### 2. 品牌识别追问机制

**问题**：104条need_brand=1的知识按6个品牌分（极目/极目单北斗/航天/雅迅/启明/有为），不同品牌答案不同。

**实现**：
- 命中need_brand=1知识时，若用户未指定品牌 → 追问"您的设备是什么品牌"，列出6个候选
- 用户选品牌后 → 用`category_l2`(主题)+品牌查该品牌对应知识返回
- **多轮上下文衔接**：追问时把`{type:"brand_collection", knowledge_id, category_l2}`存入会话metadata，下一轮用户回答品牌时跳过意图分类直接返回该品牌知识

**实测**：
```
用户: 怎么查SIM卡号
系统: 关于「查询终端号/SIM卡号/ID号」...请问您的设备是什么品牌?
用户: 雅迅
系统: [返回雅迅查询SIM/ID方法答案] ✅
```

### 3. 「是否对客」转人工机制

**问题**：知识库里有大量客服内部 SOP（如基础流量充值的内网链接、内部群、登记表格），这些内容不能直接回复给客户；另有需核实客户信息（车型/VIN）或商务跟进（开专票）类的知识，AI 直接答不合适。

**知识标记**：每条知识在 Excel 标「是否对客」列，导入后映射到 `auto_reply` 字段：
- `对客`（auto_reply=True）→ AI 直接自动答
- `转人工`（auto_reply=False）→ 命中后先追问、再转人工

转人工类知识同时填「答复策略」列（映射到 `transfer_prompt`），存转人工前追问语，如"请提供车架号(VIN)或ICCID号"。

**转人工前追问**：命中转人工类知识后，不立即转，而是用 `transfer_prompt` 追问收集关键信息（VIN/ICCID/车型等），复用品牌追问的 `pending_context` 多轮机制（type=transfer_collection）。用户回应后转人工，把收集到的信息带进工单给坐席，省去坐席重复询问。

**转人工工单携带完整上下文**：转人工时生成 `HandoffTicket`，`summary` 字段存该会话全部 message 拼接的完整对话上下文（不做 LLM 摘要），`collected_info` 存已收集业务信息。坐席看工单即可掌握全程，无需查 message 表。

**标记分布**（数据库实测）：dashcam 144全对客；wifi 8对客+1转人工(WF0008车型适配)；data 2对客+18转人工(内部SOP)；refueling 7对客+1转人工(JY0006开专票)。

### 4. 转人工决策引擎

6种触发条件（确定性规则，不依赖大模型）：
1. `consecutive_fail`：连续失败未解决
2. `keyword`：高风险词（投诉/退款/315/12315/起诉/维权）
3. `user_request`：用户明确要求转人工
4. `out_of_scope`：超出AI范围（修改绑定/激活设备/修改套餐）
5. `risk`：知识条目标记high风险
6. `non_customer_kb`：命中需人工处理的知识（auto_reply=False，即「是否对客=转人工」）

触发后生成工单（含完整对话上下文）。非工作时间 → 引导用户留问题描述+联系方式。

### 5. 大模型的作用边界

大模型只做**低频兜底**，不在正常主路径里做意图分类或全量知识检索：

| 环节 | 谁负责 | 大模型参与 |
|---|---|---|
| 意图分类 | `keyword_rule` + 代码默认规则 | ❌ |
| 业务域识别 | 规则服务 | ❌ |
| 知识检索（召回） | 本地向量模型 + MySQL向量表 + 内存余弦 | ❌ |
| 低置信候选重排 | 豆包大模型（仅topK候选） | 可选 |
| 知识回答内容 | 知识库standard_answer原文 | ❌ |
| 运营数据查询 | 接口实时查询(batchVehicleInfo) | ❌ |
| 转人工判断 | 关键词规则 | ❌ |

符合架构铁律：运营数据通过接口实时查询不直接给大模型，知识回答用标准答案/预润色答案而非大模型自由生成。

### 6. 回复润色策略：预润色优先 + 实时润色兜底

知识库的标准答案偏简练（如"菜单--设备操作管理--主机设备重启"），直接回复用户略显生硬。为此引入**受限润色**——大模型对答案做格式/语气优化，但严禁修改内容。

**两套润色机制**：

| 机制 | 说明 | 延迟 |
|---|---|---|
| 预润色（主） | 离线一次性对144条知识润色，结果存入 `knowledge_answer.polished_answer` 字段 | 零延迟（直接读库） |
| 实时润色（兜底） | 对没有预润色的新知识，运行时实时调大模型润色 | +3-5秒 |

**运行时策略**（`_polish_answer` 函数）：
```
命中知识
  ├─ 有 polished_answer → 直接返回 (零延迟, 144条都走这)
  └─ 无 polished_answer
       ├─ ENABLE_POLISH=true → 实时调大模型润色 standard_answer
       └─ ENABLE_POLISH=false → 返回 standard_answer 原文
```

**润色约束（prompt 严格控制）**：
- ✅ 允许：调整语气更友好、加项目符号/编号、分段、加引导语
- ❌ 禁止：修改/删减任何步骤、参数、数值、条件、转人工提示
- 安全兜底：润色结果比原文短一半以上（疑似丢内容）时回退原文

**效果对比**：
- 原文：`菜单--设备操作管理--主机设备重启`
- 润色：`您可以按照以下路径操作重启设备哦：\n菜单--设备操作管理--主机设备重启`

**关键参数/步骤保护验证**：U盘FAT32/16G、4G离线排查步骤①②、厂家代码72632 等关键信息润色后均原样保留。

**性能考量**：预润色把润色开销前置到一次性离线计算，运行时零延迟；实时润色仅作兜底，避免每次查询都多调一次大模型。预润色脚本：`scripts/build_polished_answers.py`（全量重算）。

### 7. 运营数据查询：接口实时查询（不落库）

**方案演进**：原计划导入运营平台设备数据到 `operational_device` 本地表，通过 SQL 查询。当前已改为**接口实时查询 + 保留本地 `operational_data` 辅助数据**。

**为什么改用接口**：
- 数据实时性：接口返回最新数据（在线状态、到期时间等），本地表会过期
- 维护成本：不再维护 `operational_device` 旧缓存表
- 架构清晰：业务数据留在运营平台，客服系统只做查询消费方

**实现**（`platform_client.py` + `database_query.py` + `chat.py`）：
```
用户问"我的车LFNAHUPMXT1E19383到期了吗"
  → 意图分类: live_query
  → 匹配查询类型 + 从消息提取VIN (正则17位, 排除I/O/Q)
  → 有VIN → 调 batchVehicleInfo 接口 (POST {vinList:[VIN]})
  → 字段映射 (carNumber→车牌号, recorderId→终端号, validDateEnd→到期)
  → 返回车牌/设备号/到期时间
无VIN时 → 追问"请提供您的车架号(VIN)"
```

**接口约束**：
- 仅支持 VIN 查询（不支持车牌/终端号反查），查询入口已统一只收 VIN
- 仅返回 4 字段（vin/车牌/设备号/到期），在线状态/SIM/套餐/服务商等待补其他接口
- 返回字段经 `field_dictionary` 的 `can_show_customer` 过滤后才展示给用户
- 接口失败/查无结果 → 提示"未查询到，请确认VIN或联系人工"
- 通过 `PLATFORM_API_URL` 配置切换线上/内网测试地址

**数据库相关变更**：
- 本地运营平台数据继续保留在 `operational_data`，当前26696行，用于接口字段不全时的辅助查询/品牌识别。
- `operational_device` 代码层依赖已移除，并已执行 `DROP TABLE IF EXISTS operational_device`，删除前0行。
- `device_vehicle_relation` 目前为空表，仅ORM兼容保留，后续确认无规划后再清理。

### 8. 品牌识别双因子：VIN + 终端号联合识别

**问题**：仅通过 VIN 查 `operational_data` 表的 `device_brand` 字段判断品牌可能出错（运营数据本身有误标）。需要用**行车记录仪ID（终端号）**做二次验证。

**终端号品牌规则**（来自业务方确认，见「有为设备10010台明细.xlsx」的「通过ID判断设备品牌」Sheet）：

| 序号 | 品牌 | 终端号规则 | 置信度 |
|------|------|-----------|--------|
| 1 | 有为 | 查 `youwei_device` 表（10010台全量），命中即确定 | 0.95 |
| 2 | 雅迅 | 00开头纯数字 | 0.95 |
| 3 | 启明 | 31开头纯数字 | 0.95 |
| 4 | 极目 | 7位纯数字(95%) / 第1或2位是A其余数字(5%) | 0.95/0.90 |
| 5 | 航天 | 数字字母混合(95%) / 90或91开头纯数字 | 0.85/0.80 |

**实现**（`brand_service.py`）：

```
identify_by_vin(VIN)
  ↓
查 operational_data → device_brand + recorder_id(行车记录仪ID)
  ↓
① 先用 recorder_id 跑 identify_by_terminal_id (终端号规则，更可靠)
  ├── 命中 + VIN结果一致 → 置信度 +0.05，path="vin_terminal_combined"
  ├── 命中 + VIN不一致 → 终端号优先，标记差异（"VIN结果与终端号不一致，以终端号为准"）
  └── 未命中 → 回退VIN查表
       ├── device_brand="鱼快" → youwei表二级区分极目/有为
       └── 其他 → _map_vin_brand 直接映射 (置信度 0.90，标记"无终端号验证")
```

**关键设计**：终端号规则优先于 VIN 查表，因为 VIN 可能出错但终端号规则是业务方确认的确定规则。

### 9. VIN主动品牌识别（无pending上下文）

**问题**：用户直接发纯 VIN（如 "LFNAHUPMXT1E19383"）时，没有 pending 上下文，意图分类可能误判为 unknown，知识检索也匹配不到，最终走到兜底回复。

**解决**（`chat.py` Step 3.5）：在意图识别之前，增加 VIN 主动品牌识别：

```
用户消息
  ↓
Step 3.5 (NEW): dashcam业务 + 无pending + 纯VIN消息
  ├── 调用 identify_by_vin (含终端号双因子)
  ├── 识别到品牌 → 返回"已通过VIN识别到您的设备品牌为「XX」"，存pending供后续复用
  └── 未识别 → 静默降级，继续走 Step 4 意图识别
```

**触发条件**（全部满足）：
- `business_area == "dashcam"`
- 无 pending 上下文
- 消息是**纯 VIN**（`stripped == vin`，避免误拦截"查设备 LFNAHUPMXT1E19383"这类完整查询）

**特点**：不依赖 pending 上下文，即使会话变了也能工作。

### 10. 向量语义检索：本地Embedding模型 + 内存召回

**问题**：之前的"大模型全量检索"每次把全量知识库（144条）发给豆包选 top5，慢（+3-5秒）且贵（每次几千 tokens）。

**方案**：用本地 `BAAI/bge-small-zh-v1.5` 模型把知识条目预计算为 512 维向量，启动时加载到内存，用户问题同模型向量化后做余弦相似度召回。

**实现**：

```
启动时
  ↓
load_to_memory(): 从 business_knowledge_embedding 表读172条向量到内存
  ↓
用户问题 → encode(query) → 与内存向量矩阵做 dot product
  ↓
topK 召回 → 按分数阈值分流:
  ├── top1 ≥ 0.62 + margin ≥ 0.05 → 直接采纳 (vector)
  ├── 0.55 ≤ top1 < 0.62 → 返回本地低置信候选 (vector_low) 或可选LLM候选重排
  └── top1 < 0.55 → 返回无命中，由上游澄清/连续失败/转人工策略处理
```

**新检索链路**（`knowledge_service.py`）：

```
L1 精确匹配 → L2 向量召回 → L3 低置信LLM候选重排 → 无命中/澄清
```

**关键参数**（`.env` 可调）：
| 参数 | 值 | 说明 |
|------|-----|------|
| `VECTOR_ACCEPT_SCORE` | 0.62 | 高于此分+margin可直采 |
| `VECTOR_LOW_SCORE` | 0.55 | 低于此分转关键词 |
| `VECTOR_MARGIN` | 0.05 | top1与top2的最小分差 |
| `ENABLE_LLM_RERANK` | true | 低置信时是否调LLM重排；关闭后仍可返回本地低置信候选 |

**source_text 构建**：拼接 standard_question + category + manufacturer + common_phrasings + variants + keywords，**不拼完整答案**，避免答案中"请先检查SIM卡"等泛化表述污染检索语义。

**性能**：向量召回 + 余弦计算 < 50ms（512维×172条），远快于之前的 LLM 全量检索（3-5秒）。

### 11. 意图识别规则化

**问题**：之前每条消息调豆包做意图分类（knowledge_query / live_query / unknown），增加 2-3 秒延迟，且对简单问法也可能分错。

**方案**：用规则替代大模型做意图分类。`_should_route_live_query()` 函数判断用户是在问**教程**（走知识库）还是**查本人数据**（走运营平台接口）。

**规则**：

```
含 VIN → live_query
含个人标记 ("我的"/"帮我查"/"到期了吗"/"在线吗"...) → live_query
含教程标记 ("怎么"/"如何"/"方法"/"步骤"...) → knowledge_query
QRY009 (续费/缴费) → knowledge_query (强制走知识库)
其他 → knowledge_query (默认)
```

**效果**：每条消息省 1 次 LLM 调用（约 2-3 秒），意图分类延迟从 2-3 秒降到 < 1ms。
