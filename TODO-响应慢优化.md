# 待办：响应慢优化

## 问题描述
用户发一条消息，后端响应需要 7-13 秒（极端情况 60-100 秒），体验卡顿。

## 根因
每条消息**串行调用 2 次火山方舟大模型**：

| 步骤 | 调用 | 耗时 |
|------|------|------|
| 1. 意图分类 | 豆包 API（classify） | ~2-3秒 |
| 2. 知识检索 | 豆包 API（retrieve，144条知识全塞prompt） | ~5-10秒 |
| 3. 品牌追问/返回答案 | 本地 SQL | <0.1秒 |

最慢的是第2步——把 144 条知识（约 3000 token）全部塞进 prompt 让大模型选，每次查询都重新发送这 3000 token。

## 可选优化方案（按推荐度排序）

### 方案A：意图分类改回关键词规则（推荐，改动小见效快）
- 意图分类不再调大模型，用关键词规则判断（转人工/问候/查询/知识问答）
- 只在知识检索这一步调大模型
- **效果**：省一次 API 调用，快 3-5 秒
- **代价**：意图识别变粗，口语化问题可能误分类（之前正是因为规则太粗才换的大模型）
- **折中**：规则优先，规则判断不准时再调大模型兜底

### 方案B：改用向量检索（最快，但需解决网络问题）
- 144 条知识预先算好向量缓存到内存
- 查询时只编码用户问题（1次小API）+ 内存算相似度
- **效果**：检索从 5-10秒 降到 1-2秒
- **障碍**：之前本地 sentence-transformers 因 huggingface 网络问题没跑通；火山方舟 embedding 模型需要单独开通
- **前提**：需先开通火山方舟 embedding 模型，或解决 huggingface 镜像连接

### 方案C：只调参数，不改架构（提升有限）
- 大模型调用加超时限制（避免 60 秒极端情况）
- 降低 max_tokens
- 开启流式输出（用户能更快看到首字）
- **效果**：缓解极端慢，但平均 7-13 秒降不下来

### 方案D：合并意图分类+检索为一次调用
- 让大模型一次性完成"意图分类 + 知识检索"
- **效果**：2次API合并为1次，快 3-5 秒
- **代价**：prompt 更复杂，大模型可能两头都做不好；调试更难

## 当前状态

- 后续优先考虑方案A（最简单）或方案B（最彻底）
